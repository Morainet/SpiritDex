## 背景与洞察

探索确认：**前端、后端表结构、entity、DTO、Service 全部已为「完整技能池」做好预留**（`pet_skill` 表有 `learn_method`/`unlock_level` 字段、DTO 是 `List<PetSkillDto>`、前端 `SkillTable` 连 4 种学习方式中文标签都写好了）。瓶颈只在三处：①scraper 还没抓 native/stone/blood 技能（数据源结构未经实测）；②后端 `SeedRunner` 漏读 `unlock_level`；③文档漂移。

本轮按你的选择：**先探路验证数据源** + 修复文档漂移 + 后端补 unlock_level 读取 + 验证活动抓取链路。技能池的全量抓取解析器在探路明确结构后再实施。

## 任务 1：探路——验证 BWIKI 精灵页面技能池数据结构（核心前置）

**目标**：实测「喵喵」等 2-3 只精灵的页面级 wikitext，搞清 native/stone/blood 技能到底在哪、什么格式。这是后续写解析器的地基。

**实现**：写一个临时探路脚本（放在 `scraper/scripts/probe_pet_skills.py`），复用现有 `WikiApi.fetch_module_wikitext(page)`（api.py:52，已支持任意 page 名，不限于 `Module:` 前缀）：
- 抓「喵喵」「水蓝蓝」「火花」三只御三家页面 wikitext
- 全文打印前 3000 字符 + 检索关键字（`技能`/`skill`/`秘籍`/`血脉`/`天赋`/`等级`/`{{`模板名）
- 定位技能池载体：是 `{{精灵技能表|...}}` 模板？还是 `#invoke` Lua 调用？还是另一个 Module？
- 把原始 wikitext 存到 `scraper/tests/fixtures/page_喵喵.txt` 等留档

**产出**：一份探路报告（输出到对话），明确：(a) 技能池数据的真实结构；(b) 推荐的解析策略（正则 / 模板参数解析 / 新 Module 抓取）；(c) 估算可抓取覆盖率。**基于探路结果再决定任务 4（全量抓取）的具体实现**。

**风险预案**：若三只页面结构不一致、或技能池根本不在 wikitext（而是动态渲染），则在报告中说明并调整后续方案。

## 任务 3：后端补 unlock_level 读取（一行遗漏修复）

**位置**：`spiritdex-api/src/main/java/com/spiritdex/api/seed/SeedRunner.java` 第 214-219 行

当前 `seedPetSkills` 方法：
```java
PetSkill ps = new PetSkill();
ps.setPetId(petId);
ps.setSkillId(skillId);
ps.setLearnMethod(str(it.get("learn_method")));
ps.setDeleted(0);
petSkillMapper.insert(ps);
```

**修复**：在 `setLearnMethod` 之后补一行读 `unlock_level`：
```java
ps.setUnlockLevel(intOrNull(it.get("unlock_level")));
```
（`intOrNull` 工具方法已在第 396 行存在，当前 JSON 里 `unlock_level` 全是 null，此修复让字段真正贯通，为技能池数据落地做准备。）

同时检查 `PetSkill.java` entity 确认有 `unlockLevel` 字段及 setter（探索报告确认有，快速核对）。
