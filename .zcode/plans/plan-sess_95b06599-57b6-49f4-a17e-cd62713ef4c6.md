# 定时编写发布最新活动攻略功能 — 实施计划

## 目标
每周一定时从 BWIKI 拉取最新活动公告 → 调智谱 GLM 扩写成结构化活动攻略 → 落库为 `draft` → 次日定时自动转 `published` → 前端 ISR 实时可见。全链路零账号依赖，无 GLM key 时优雅降级。

## 架构链路
```
@Scheduled(周一8点) ──► scraper抓BWIKI活动 ──► GLM扩写攻略 ──► article表(draft, ai_generated=true)
                                                                          │
@Scheduled(周二8点) ──► draft→published + 生成embedding索引 ◄──────────────┘
                                                                          │
前端 app/articles/[slug] ◄── ISR revalidate=3600 + dynamicParams=true ◄────┘
```

## 改动清单（5 个模块）

### 1️⃣ 数据库迁移 `V9__article_ai_generation.sql`（新增）
给 `article` 表加两列，**不动现有数据**：
- `ai_generated BOOLEAN NOT NULL DEFAULT false` — 标识 AI 生成（前端可显示「AI 生成」徽章 + 合规）
- `source_url VARCHAR(512)` — 活动来源 BWIKI 链接（溯源）

### 2️⃣ scraper 扩展（新增 `src/activity_fetcher.py` + 导出）
- 复用 `src/api.py`（已有 MediaWiki API 封装、QPS≤1、UA、重试）
- 新增函数拉取 BWIKI 首页 wikitext → 用现有 `lua_parser` 风格的轻量解析提取 `{{活动公告|...}}` 模板参数（活动名/时间/链接）
- 输出 `data/seed/activities.json`（含 `name/start/end/source_url/raw_text`，带 `scraped_at`/`source`）
- `main.py` 加 `--activities` 开关；若 BWIKI 模板结构不符预期，回退用 `api.php?action=query&list=search&srsearch=活动` 拿候选页
- **诚实说明**：BWIKI 活动 wikitext 结构未实测，解析逻辑会写成「尽力提取 + 兜底 raw_text」，必要时退化为「拉首页全文交给 LLM 自己摘取活动段落」，保证不阻塞主流程

### 3️⃣ 后端 AI 攻略生成（`spiritdex-api`，核心）
**新增文件：**
- `ai/ArticleGenerationService.java` — 核心生成服务
  - 读 `activities.json` 取最近 N 条活动
  - 每条活动调 `chatModel.call(prompt)`（阻塞式，仿 `IdentifyService.callWithRetry` 的 3 次重试退避）
  - 解析 GLM 返回的结构化结果（标题/摘要/tags/正文 Markdown）
  - slug 生成：`event-{yyyyMMdd}-{活动名拼音或hash前8位}`，查重保证 UNIQUE
  - 按 slug upsert 到 `article` 表，`status="draft"`、`ai_generated=true`、`author_name="灵宠档案 AI 助手"`、`category="活动"`
- `ai/ArticlePublishService.java` — 草稿转发布
  - 查 `ai_generated=true AND status='draft' AND created_at < NOW()-24h` 的文章
  - 改 `status="published"`
  - **顺带建 embedding 索引**：调 `EmbeddingService.embed(content)` → `EmbeddingMapper.insertVector("article", id, slug, content, vec, "bge-small-zh")`（让 RAG 问答能召回文章）

**扩展 `Prompts.java`**（加常量，仿现有风格）：
- `ARTICLE_SYSTEM` — 攻略写作人设（基于真实活动数据，不编造数值，Markdown 输出，结构化字段约定）
- `ARTICLE_USER_TEMPLATE` — `{activity_info}` + 现有精灵/相克知识上下文 + 输出格式要求（title/summary/tags/content 的分隔符约定，便于解析）

**配置：** 在 `application.yml` 的 `spiritdex.ai` 段下加：
```yaml
article-generation:
  enabled: false              # 默认关，生产显式开
  generate-cron: "0 0 8 ? * MON"   # 每周一8点
  publish-cron: "0 0 8 ? * TUE"    # 每周二8点
  draft-age-hours: 24
  max-articles-per-run: 3
```
绑定到 `AiProperties` 的内嵌静态类 `ArticleGeneration`。

### 4️⃣ 调度基础设施（项目首次引入）
- `SpiritdexApiApplication.java` 加 `@EnableScheduling`
- 新增 `schedule/ArticleSchedule.java`（`@Component` + `@Scheduled`），两个方法：
  - `generate()` — 调 `ArticleGenerationService.generateLatest()`，cron 从配置读
  - `publish()` — 调 `ArticlePublishService.publishDueDrafts()`，cron 从配置读
- **关键决策**：用 `@ConditionalOnProperty(name="spiritdex.ai.article-generation.enabled", havingValue="true")` 而非 `@Profile` —— 因为生产 Docker 默认 profile 跑，加 `@Profile` 会导致定时任务在生产不生效（探索发现这是项目现有 CommandLineRunner 的痛点）
- 守卫：每个任务开头 `if (!aiProperties.isEnabled()) return;`（无 GLM key 自动跳过，复用三个 Seeder 的模式）

### 5️⃣ 管理端点（可选但推荐，便于手动触发/调试）
`controller/AdminArticleController.java`（`@RequestMapping("/api/admin/articles")`）：
- `POST /generate` — 手动触发生成（传 body 可覆盖活动源）
- `POST /publish-due` — 手动触发到期草稿转发布
- `GET /drafts` — 列出当前 draft
> 注：当前项目无鉴权（Phase 7 才接 Spring Security），先裸暴露，上线前用 Nginx 加 IP 白名单或 basic auth 兜底

### 6️⃣ 前端适配（ISR + AI 标识）
- `app/articles/[slug]/page.tsx`：
  - `export const dynamicParams = false` → `true`
  - 加 `export const revalidate = 3600`（每小时 ISR，新发布文章 1 小时内自动可见）
- `app/articles/page.tsx` 列表页：列表卡片若 `aiGenerated` 为 true 显示小徽章「AI」（合规透明）
- `types/article.ts`：`ArticleListItem`/`ArticleDetail` 加 `aiGenerated?: boolean`、`sourceUrl?: string`
- `ArticleListItemDto.java`/`ArticleDetailDto.java`：加对应字段

## 关键技术决策
1. **draft + 24h 延迟发布**：AI 偶发幻觉有 24h 缓冲，期间可人工在 Swagger 改 `status`/删草稿
2. **`@ConditionalOnProperty` 而非 `@Profile`**：生产 Docker 用默认 profile，这样定时任务生产能跑（探索发现的现有坑）
3. **slug 规则 `event-{日期}-{活动标识}`**：稳定、唯一、URL 可读，与现有 `pet-{dex_no}` 风格一致
4. **embedding 在发布时建而非生成时建**：draft 不进检索库，避免半成品污染 RAG
5. **配置默认 `enabled: false`**：避免开发环境意外跑、意外烧 GLM token；生产用环境变量 `SPIRITDEX_AI_ARTICLE-GENERATION_ENABLED=true` 开启
6. **BWIKI 解析容错**：activity_fetcher 写成「优先结构化解析，失败兜底传 raw_text 给 LLM 自摘」，保证抓取失败不阻塞生成流程

## 验证方式
- 后端单测：MockGLM 返回固定 Markdown，断言 ArticleGenerationService 解析正确 + slug 查重
- 手动：`SPIRITDEX_AI_ARTICLE-GENERATION_ENABLED=true` 启动后端 → `POST /api/admin/articles/generate` → 查 DB 有 draft 文章
- 前端：`pnpm dev` → /articles 列表「活动」分类可见 AI 文章 → 详情页 ISR 生效
- 全链路：调 `publish-due`（或临时改 `draft-age-hours: 0` 测试）→ 文章转 published + embedding 表新增 article 记录
